## 🛑 血的教训（2026-06-06 更新）

### 深度解读先行原则（2026-06-06 新增）
任何产品方向必须先做深度解读（市场分析+前景判断+机会评估），静渊判断完说"做"才做，说"先别搞"就停。

### 记忆必须写文件（2026-06-06 新增）
每次会话中有决策/结论/教训，立刻写入MEMORY.md或memory/当日文件。不在会话里说"我记住了"——这句话没有任何用，因为下次会话醒来文件才是记忆。

### Cron 跑完必须留底（2026-06-06 新增）
cron 跑完的内容（尤其是早报）不能只发微信不留底。必须同步写入 drafts/ 目录存档，方便以后复盘。已修改 tech-pulse-cn prompt 加入这个要求。

### 函数签名变更 = 全量 grep
改了函数签名（showFB(id,text,good)→showFB(id,idx)），必须 grep 所有调用点。
不改全 → 参数错位，反馈文本显示为索引值"1"。

### 补丁阈值=3
同一个文件补丁超过 3 次 → 暂停，评估是否该整体重写。
补丁堆叠代码会"乱套"，最终还是要重写，更耗时。

### 工具选型
- 复杂 JS 字符串：用 `edit` 工具精确匹配，不用 sed（转义引号会崩）
- Python 批量替换后：必须检查 JS 语法，容易留 `,,` 双逗号

### 镜像修复
改了 A 页面的 bug → 顺手检查 B、C 页面有没有同类问题。
kh 从 speechSynthesis 切到 MP3 → cp 和 al 也得检查。

### 主动监控Cron投递状态
tech-pulse-cn 早报（6:30触发）每天跑完后必须检查 `delivered` 字段。如果 `delivered: false`，立刻手动重跑（`cron run`），不等静渊来问。
已踩坑：6/4超时未交付、6/5投递失败未主动发现。

### 交付前全流程测试
修改后必须走完整用户路径（首页→...→完成），不能只测改的那块。
用户两次批评"你做完了能不能自己检查一遍"。

### 音频覆盖全量检查
加音频时不能一个一个补——先 grep 所有场景，列清单，一次性补全所有缺音频的场景。
用户发现某一页没声音时，说的是"从此页往后都没声音"，实际意思就是后面全缺，必须一次修干净。

### 项目边界=不可逾越
燃气险和书游时代是**两个独立项目**，各有各的诉求和用户。永远不能混到同一个导航页/卡片组里。
切换项目时先确认用户说的是哪个项目的事，别把A的改动带到B。

### 域名DNS配置后告知清单
配置完域名/DNS/Cloudflare后，必须一次性告知用户：
1. 可能需要等几分钟生效
2. 电脑上 `ipconfig /flushdns` + 重启浏览器
3. 手机上开飞行模式5秒再关，或切5G
4. 还是不行就用 IP 直连备用

### API 卡壳30分钟 = 换方案
任何 API 踩坑（Cloudflare 等），30分钟搞不定就换方案。
凌晨不做复杂技术部署，判断力下降容易绕死胡同。

### 不要让用户动手
让用户下载东西→手动上传 = 承认自己搞不定。特别是一次36MB的大文件上传。
遇到用户搞不定的步骤，自己先试通再告诉用户怎么做，而不是让用户当小白鼠。

## 移动端音频关键原则（2026-06-06 更新）
- 所有 `play()` 必须在用户点击的**同一帧**执行（onclick链内）
- **绝对不能**用 `setTimeout` 包裹 `play()` — 浏览器会阻止
- 但同一个 onclick 链中，先播音频再切页面（例如 `setTimeout(go, 500)`）没问题
- 每个场景都要有音频：`sceneMp3`（MP3文件）或 `playMp3()` 调用，缺一不可

## 带宽/并发经验（2026-06-05 更新）
- 轻量服务器1-2Mbps带宽
- 纯静态页面（无音频）：50-100人同时在线
- 用户边看边听配音（50-70KB/文件）：20-30人
- 全部开BGM+配音（BGM 2.1MB流式播放）：8-15人
- 瓶颈是带宽，不是CPU/内存（4核/3.6G够用）
- 音频放CDN是最直接的扩容方案
