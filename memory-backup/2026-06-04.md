## 一、燃气险培训工具（产品宣传/话术/案例）

### 教训 #1：先问清楚架构再动手

上午直接做了8个模块合一的 `train.html`，用户看了说"理解错了"。他要的不是一个页面走到底的培训工具，而是**三个独立页面**，每页5-10分钟，各自聚焦一个主题：
- ① 产品宣传（ABCD款/保障范围/保险标的）
- ② 销售话术（开口7句话/三段话术）
- ③ 理赔案例（真实案例/客户问答）

**教训：** 用户说"做燃气险培训工具"时，应该先画出3个框（产品/话术/案例），确认结构再动手。直接做8个模块浪费了时间。

### 教训 #2：用户说"发材料"不等于"发完了"

用户说有两个视频要发，我说"发过来"，他发了。但实际素材远不止视频——还有火灾现场照片、对比图、风险告知图等。主动问用户"你手上还有什么材料"比被动等发更快。

### 行动项
- [ ] 等用户整理好材料后重新做三个页面
- [ ] 图片要直接用素材里的实拍照片（不是网图）

## 二、圯桥拾履互动页（书游时代）

### 经验 #1：浏览器语音合成不靠谱

Web Speech API (`speechSynthesis`) 在微信浏览器和 Safari 上经常不出声。解决方案：用 `edge-tts` 工具批量生成 MP3 文件，直接嵌入 `<audio>` 标签播放。

18个场景的MP3（~1.2MB总量）用 `zh-CN-XiaoxiaoNeural` 女声生成，效果稳定。

### 经验 #2：微信/移动浏览器阻止自动播放

用户必须手动点击一次页面（比如点击"开始体验"按钮），才能解锁音频播放。这是一个iOS/Android的安全限制，无法绕过。

### 教训 #3：逐字打字动画容易和点选逻辑冲突

今天试了三次加入逐字动画：
1. 第一次：`setTimeout(doNextChar, 18)` 的递归链和 `pick()` 同时跑，旧的动画链干扰新场景的渲染
2. 第二次：加了 `showButtons()` 回调，但旧链未清理导致按钮提前出现
3. 第三次：回退到无逐字动画的稳定版本

**教训：** 交互逻辑和动画逻辑要解耦。简单的CSS淡入+按钮弹入已经够好看，没必要用JS控制逐字渲染引入竞态条件。如果要加逐字效果，应该用CSS动画或独立的 Web Worker，不与交互流混在一起。

### 教训 #4：getElementById('footer') 写错class

JS代码里写了 `document.getElementById('footer')`，但HTM L元素是 `class="footer"` 没有id。这行代码在旧版本页面里没有（之前版本可以正常点），但在重构时加了这行，导致点击开始没反应。

**教训：** 重构时引入的每一行新代码都要在浏览器实测。getElementById 对不存在的元素返回 null，访问 null.style 直接报错停掉整个函数。

### 经验 #5：用户的朋友给出了高质量反馈

用户朋友试玩后反馈：
- "要有动画画面感" → 加了背景图
- "要有声音" → 加了 MP3 旁白 + 古筝背景音乐
- 这些反馈比用户自己提的"想要什么"更具体、更可执行

### 经验 #6：背景图一张就够

用户发了9张图片，最后说"就这张够了吧"——一张黄石公坐山巅的水墨图，封面和故事背景通用。少即是多，一张好图胜过九张轮播。

### 经验 #7：npx serve 比 Python http.server 稳定

Python `http.server` 会在后台静默崩溃。`npx serve -l 8080` 的 301 重定向问题（如 `/play.html` → `/play`）可以通过 `curl -sL` 跟随重定向解决，浏览器也会自动跟随，所以对用户无影响。

### 经验 #8：给用户发横屏图片要小心token消耗

用户一直发图片（第一天发现自己可以发图了，很兴奋），每张都256K-1MB。如果每一轮都发图，token会快速烧完。建议策略：多轮对话中，只在前两轮详细描述图片，后续仅记录文件名，不重复描述内容。

## 六、产品页重构（cp.html）教训

### 教训 #1：先拿原件再干活

做 cp.html 时凭记忆写 ABCD 数据，C款总保额写成了 903万。实际投保单上是 **795万**。

**教训：** 源文档数据必须第一个提取，数据以原件为准，不靠记忆。

### 教训 #2：图片不能猜，得确认

从docx抽了4张图，自己猜哪张是投保单、哪张是保险责任，结果放反了。用户一眼看出来。

**教训：** 图片标注清楚，分不清的直接问用户，不猜。

### 教训 #3：结构按文档走，不加不减

文档规定流程是：产品介绍→保险责任→保单获取→交互体验。我跳过了保险责任直接到问答，用户纠正了。

**教训：** 文档规定的流程就是产品逻辑，读者需要按顺序学。

### 教训 #4：命名要全称

写"A款"没问题，但用户要求"宜家安馨—福A款"。

**教训：** 正式产品名一个字不能省。

### 教训 #5：每页只做一件事

投保单页面不该混入保险责任图片。用户说"投保单就一张照片"。

**教训：** 每页一个焦点，干净利落。

### 教训 #6：先教后考

交互问答放在保险责任介绍之后。还没教就考，用户答不上来。
先学完（5小节保险责任），再来6道题巩固。

## 七、话术页（kh.html）教训

### 教训 #1：speechSynthesis 不可靠，必须用 MP3

用户反复投诉"还在跳字跳行"，说"怎么就是不改"。我一开始试图调文字、调速度，但其实根因是浏览器语音合成不可靠——长中文文本在 Chrome/微信浏览器中经常跳字跳行。

最终解决方案：用 edge-tts 生成 MP3 嵌入 `<audio>` 标签。跟圯桥拾履一样的方法，一次解决。

**教训：** 任何需要朗读长文本的场景，直接上 edge-tts MP3，不要再用 speechSynthesis 试试看。省得用户说"怎么就是不改"。

### 教训 #2：用户说"理解错了" → 做了再验证

用户发"系统显示您家还没有办理燃气保障"和"刚才在系统查到已经脱保了"两份话术时，我一开始按"老用户/新用户"理解。用户纠正：不是按用户类型分，是按**产品类型分**（一般款/特别款）。

**教训：** 用户给的文案标题就是正确分类。用户说的分类方式直接照用，别自己另起一套。
