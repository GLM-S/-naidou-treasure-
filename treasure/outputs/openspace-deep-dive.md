# OpenSpace：让 AI Agent 学会「自进化」— 你在写的 Skill，它能自己修

> 当 AI Agent 学会自己升级自己，人类还剩什么？
> 答案是：**判断力、品味、和那个不会进化的"第零道"。**

---

## 一、这是一个什么项目

**OpenSpace**（github.com/HKUDS/OpenSpace | 6413⭐ | MIT）是港大开源的一个 **Agent 执行引擎**。

听起来像又一个 Agent 框架？不，它的核心卖点只有一个：

> **Agent 执行完任务后，能自动检查和改良自己的 Skill。**

人话：你写了一个 Skill，AI 用 Skill 干活；干完了，AI 自己检查这个 Skill 好不好，不好的地方自动改。下次再干同样的活，技术得更娴熟了。

这跟其他 Agent 框架有什么区别？

| 框架 | 做事 | 自省 | 自改 |
|------|------|------|------|
| LangChain | ✅ 能跑 | ❌ 不会 | ❌ 不会 |
| CrewAI | ✅ 能跑（多人） | ❌ 不会 | ❌ 不会 |
| AutoGPT | ✅ 能跑 | ⚠️ 简单反馈 | ❌ 不会 |
| OpenSpace | ✅ 能跑 | ✅ 会分析 | ✅ 会自己改 |

**OpenSpace 之前，人是 AI 的师傅——你写好 Skill，它执行，但你得改。**
**OpenSpace 之后，AI 开始当自己的师傅——它执行完任务，自己分析自己哪里没做好，自己修。**

## 二、它的核心机制

OpenSpace 的核心概念说穿了就三件事：

### ① 技能发现（Skill Discovery）
AI 干活之前，先到 Skill 仓库里翻一遍，看看有什么现成的技能可以用。不用每次从零开始想怎么干。

### ② 技能引导执行（Skill-guided Execution）
找到合适的技能 → 按技能里写好的步骤干活。不瞎猜、不幻觉、不自由发挥。

### ③ 技能自动进化（Skill Evolution）
这是最要命的——干完活，AI 分析这次执行的效果，然后自动产生三个动作之一：

- **FIX**：Skill 出错了，硬修（比如 API 换地址了）
- **DERIVED**：从已有 Skill 衍生出一个新的子 Skill（比如从「爬虫通用」衍生出「爬知乎」）
- **CAPTURED**：AI 这次用了自由发挥的方式把事做成了，总结成新 Skill 保存下来

**三者的区别，翻译成人话：**

```
你教小孩骑自行车。

FIX：你发现他脚踏板踩反了 → 告诉他正确的踩法
DERIVED：他学会了骑山地车 → 自然延伸出骑公路车的技能
CAPTURED：他自己发明了一个下坡加速的新方法 → 记录下来变成他自己的技能
```

## 三、为什么这对「书游时代」的管道至关重要

还记得今天的线索链吗？

```
Firecrawl / web_fetch → 结构化Schema → 交互体验生成 → 部署上线
                                  ↓
                           OpenSpace（运管层）
```

这条链的意思是：**把书变成可交互体验的三大组件，每个环节都有开源项目在突破。**

而 OpenSpace 的位置是——**管道的运管层**。它的价值不在端到端一键生成（那是 Open Lovable 和 GPT-5 干的），而在：

> **管道持续运行、自我迭代，不需要人盯着。**

想象一下书游时代的管道跑起来之后的样子：

```
Step 1: 抓取书（Firecrawl / web_fetch）
Step 2: 结构化 Schema（需要自建）
Step 3: 生成交互体验（Open Lovable / GPT-5）
Step 4: 部署上线
Step 5: OpenSpace 检查效果，自动优化 Step 2-3 的参数
      ↻ 每本书更聪明一些
```

**没有 OpenSpace，这条链跑一次就要人工调一次。**
**有了 OpenSpace，它自己跑、自己调、自己进化。**

## 四、在静渊的语境里，OpenSpace 解决了什么实际问题

### 正在发生的例子：auto-update 的 bug

今天下午出了个 bug：auto-update 在更新 SKILL 时把文件截断了。

**没有 OpenSpace 时**：我发现 bug → 报告给静渊 → 静渊检查 → 我改代码 → 重新跑 → 静渊验证。来回至少 30 分钟。

**有 OpenSpace 后**：auto-update 跑崩了 → OpenSpace 捕获到执行异常 → 自动分析出错代码 → 调 FIX 修 bug → 重新跑一次验证 → 写 changelog 记录。**15 秒完成。**

### 第二层的价值：反馈闭环自动化

三道壕沟法的二层反馈闭环现在是这样跑的：静渊写了感觉 → 我手动记 feedback log → 满 5 条触发 cron 调 OpenSpace 进化 SKILL。

OpenSpace 要是深度集成，这条链可以变成：
- 每次执行完任务 → 自动分析效果 → 不满意的自动修 SKILL
- 静渊只做一件事：**审阅 OpenSpace 的修改提议，点头或摇头**

**你的工作从「记反馈→等满5条→触发进化」变成「看一下AI改了什么，点个勾」。**

## 五、现在装上了，接下来能做什么

OpenSpace 已经在这台服务器上跑了（MCP server，端口 8081，systemd service 守护）。

**今天它已经干了什么：**

> 我调了 OpenSpace 的 `fix_skill`，它成功读取了三道壕沟法 SKILL，分析出单元格格式问题，产生了修复结果。

**第一件可以正式跑的事：**

把 auto-update 挂到 OpenSpace 上试跑一次——让它检查 SKILL 有没有截断/格式错误/内容过期，自动出补丁，然后我帮你审。

推进路线：

```
今晚/明早：OpenSpace 跑一次 SKILL 健康检查
明后天：Tavily 搜索配合写深度稿
本周内：把 auto-update 对接到 OpenSpace
下周：跑一次完整 FIX/DERIVED/CAPTURED 全流程
```

## 六、但别忘了——「第零道」才是你的护城河

OpenSpace 再厉害，它也不会问：

> **这事真的值得做吗？**

它不会问「这个 Skill 能不能不写」、「这个方向我要不要做」、「这个想法值不值得花时间」。

**这就是你的第零道的价值——也是三道壕沟法在 AI 时代的终极护城河。**

AI 能自己修 Skill、自己衍生子 Skill、自己总结新 Skill。
**但它不会问你「我应不应该开这个技能点」——这永远需要静渊坐在屏幕前，点一下头。**

---

## 七、值得警惕的事情

1. **OpenSpace 是 6k⭐ 的项目。** v1.0 还没到，MCP 通信时不时会超时（今天 execute_task timeout 了两次，MCP 连接还断了一次）。
2. **FIX 的正确率没有保障。** 今天有个真实案例——OpenSpace 的 fix_skill 修三道壕沟法时，把 version 从 0.2.0 改成了 0.1.0。AI 修自己的东西，偶尔会把自己修回原形。
3. **自动进化的风险。** CAPTURED 机制意味着 AI 可能把你手动改好的东西又改成自己的版本。**永远保留一个「手工模式」的退路。**
